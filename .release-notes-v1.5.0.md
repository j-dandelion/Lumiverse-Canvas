## v1.5.0

Settings panel toggles and the Follow/Show/Hide segmented control
now actually respond to taps. The previous build silently threw a
`ReferenceError` on every tap.

### Fixed

- **Settings panel toggles and segmented control respond to taps.**
  The root cause was a set of bare cross-module variable references
  in `src/frontend.ts`. Several module-level variables
  (`DEBUG`, `_secondarySidebarOpen`, `_secondaryWrapper`,
  `_tabAssignments`) were declared in one file and referenced by
  their bare names in another, without going through the accessor
  functions that wrap them. Each tap ran the handler, hit a bare
  reference like `DEBUG = next.debugMode` or
  `if (_secondarySidebarOpen)`, the JavaScript runtime threw
  `ReferenceError: DEBUG is not defined`, the handler's `try/catch`
  caught the throw, and the toggle never flipped.

  All bare references are now routed through the accessor functions:

  | Variable | Declared in | Accessor |
  |---|---|---|
  | `DEBUG` | `src/debug/log.ts` | `getDebug()` / `setDebug()` |
  | `_secondarySidebarOpen` | `src/sidebar/secondary.tsx` | `isSecondarySidebarOpen()` |
  | `_secondaryWrapper` | `src/sidebar/secondary.tsx` | `getSecondaryWrapper()` |
  | `_tabAssignments` | `src/tabs/assignment.ts` | `getTabAssignments()` / `hasTabAssignment()` |

  The bug was equally present on desktop — nothing exercised the
  tap path until someone tried it on a phone.

- **Settings panel re-renders after the saved layout loads.** The
  settings panel was mounted before `loadSavedLayout` resolved, so
  it was built with the default `_settings` and never refreshed
  when the loaded values arrived. After this fix, the panel shows
  the saved state on every page load (so flipping "Remember
  layout" off, reloading, and reopening the panel shows it off,
  not on).

- **`getTabAssignments()` return type widened from `ReadonlyMap` to
  `Map`** to allow one caller's `.set()` in the applyLayout restore
  path. The previous type was aspirational; the only mutable caller
  is a single in-place update during layout restore.

### How this was found

Headless Chromium via Playwright with iPhone 14 Pro emulation and
`hasTouch: true`. The mobile-emulated tap on a settings toggle
fired the handler, the console logged `[pageerror] DEBUG is not
defined`, the handler's `try/catch` swallowed it, and the
`aria-checked` attribute never changed. That pageerror was the
smoking gun; the bare-reference audit followed from it.

### Files touched

- `src/frontend.ts` — replaced 16 bare cross-module references
  with accessor-function calls; added `getTabAssignments` and
  `hasTabAssignment` to the import from `./tabs/assignment`;
  added `refreshSettingsPanel()` call after the post-load settings
  hydration.
- `src/tabs/assignment.ts` — `getTabAssignments()` return type
  changed from `ReadonlyMap` to `Map`.

### Installation

```bash
bash build.sh    # bundles + copies to ~/Lumiverse/data/extensions/canvas/repo/
```

Then hard-refresh the browser (Ctrl+F5 on desktop, or clear site
data on mobile Chrome).
